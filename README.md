## 🎯 Project Overview

**EvidenceAI** is a **Multimodal Retrieval-Augmented Generation (RAG) Chatbot** designed for forensic report analysis.

### What it does:
- 📄 **Upload PDF incident reports** → Text is extracted, chunked, and embedded into ChromaDB (case-isolated)
- 📸 **Upload crime scene / evidence images** → Analyzed by **Groq Vision AI** (Llama 4 Scout) with 7-category forensic depth, then embedded
- 💬 **Ask questions** → Top-5 relevant chunks retrieved via ChromaDB (case-only search) → Answer generated by Groq (`llama-3.3-70b-versatile`)
- 🗂️ **Case Supervisor** → Auto-links uploads to cases using semantic similarity, prevents cross-incident contamination
- 🔍 **Detect inconsistencies** between report text and image descriptions
- 🎬 **Cinematic intro video** plays on load with particle burst / curtain-split exit animation
- 🌗 **Dark / Light theme toggle** in the top-right corner — pure black or clean white

### Key Features:

| Feature | Technology |
|---------|-----------|
| Text Embeddings | HuggingFace `all-MiniLM-L6-v2` (384D) |
| Image Analysis | **Groq Vision API** — `meta-llama/llama-4-scout-17b-16e-instruct` |
| Vector Database | ChromaDB (persistent, case-isolated collections) |
| LLM Inference | Groq API `llama-3.3-70b-versatile` |
| Backend | Python FastAPI (port 8000) |
| Frontend | Next.js 14 + Custom Animated Background + Tailwind CSS |

| Intro | Fullscreen `animation.mp4` with cinematic exit transition |
| **Case Supervisor** | Auto-linking uploads to cases using semantic similarity |
| **Jailbreak Guard** | XML-based prompt injection defense with regex input scanner |
| **HYDE** | Hypothetical Document Embeddings for 10-30% better recall |
| **Parent-Child** | Hierarchical chunking: fine search + full-page context |
| **RAG Evaluation** | Grounding score, retrieval relevance, logged to JSON |
| **Timeline Reconstruction** | Regex + LLM extraction of time-referenced events, sorted chronologically |
| **AI Hypothesis Generator** | ACH-inspired competing investigative theories rated by evidence strength |
| **Memory Across Sessions** | Persistent case summaries + AI-powered cross-case comparison |

## 🧠 How Generative AI Works

### What is a Large Language Model (LLM)?

An LLM is a neural network trained on massive text data to predict the next word/token in a sequence.

**Training process:**
```
Input:  "The forensic team found a ___"
Model:  Predicts → "knife" (highest probability token)
```

### The Transformer Architecture

All modern LLMs (GPT, Llama, Mistral) are based on the **Transformer** architecture (2017).

```
Input Tokens → Embedding Layer → [Transformer Block × N] → Output Probabilities

Each Transformer Block contains:
┌─────────────────────────────────┐
│   Multi-Head Self-Attention     │  ← Tokens "look at" each other
│             │                   │     to understand context
│             ▼                   │
│       Layer Normalization       │
│             │                   │
│             ▼                   │
│     Feed-Forward Network        │  ← Non-linear transformation
│             │                   │     for feature extraction
│             ▼                   │
│       Layer Normalization       │
└─────────────────────────────────┘
```

### Self-Attention Mechanism

Self-attention is the KEY innovation. It allows each word to "attend to" every other word.

**Example:**
```
Sentence: "The knife was found in the kitchen near the broken window"

When processing "knife":
  - High attention → "found"          (what happened to it)
  - High attention → "kitchen"        (where it was)
  - High attention → "broken window"  (related evidence)
  - Low attention  → "The"            (not important)
```

**Math behind attention:**
```
Attention(Q, K, V) = softmax(QK^T / √d_k) × V

Where:
  Q = Query matrix  (what am I looking for?)
  K = Key matrix    (what do I contain?)
  V = Value matrix  (what information do I carry?)
  d_k = dimension of keys (for scaling)
```

### Tokenization

LLMs don't process text directly — they process **tokens**.

```
"forensic investigation" → ["for", "ens", "ic", " invest", "igation"]
                              ID: [1234, 5678, 234, 8901, 3456]
```

### Temperature

Controls randomness in generation:
- **Temperature = 0.0**: Always picks highest probability token (deterministic)
- **Temperature = 0.3**: Slightly creative — we use this for forensic accuracy
- **Temperature = 1.0**: Very creative (bad for factual answers)

## 🔥 How RAG Works (Deep Dive)

### The Problem RAG Solves

**Without RAG:**
```
User: "What weapon was found in the incident?"
LLM:  "Based on my training data, common weapons include..."  ← WRONG! Generic answer.
```

**With RAG:**
```
User: "What weapon was found in the incident?"
RAG:
  1. Search ChromaDB → Find chunk: "A kitchen knife with blood traces was recovered..."
  2. Inject chunk into prompt
  3. LLM: "According to the incident report, a kitchen knife was found..."  ← CORRECT!
```

### RAG = Three Steps

```
R (Retrieval)  → Find relevant information from your data
A (Augmented)  → Add that information to the LLM prompt
G (Generation) → LLM generates answer using that information
```

### Step 1: Document Ingestion (Indexing Phase)

```
                    INDEXING PIPELINE

PDF File ─────────────────────────────────┐
  │                                        │
  ▼                                        │
PyPDF: Extract text per page              │
  │                                        │     Image File
  ▼                                        │         │
Text: "On January 15, officers             │         ▼
responded to a burglary..."               │  Groq Vision API
  │                                        │  (Llama 4 Scout)
  │                                        │  Analyzes: scene overview,
  │                                        │  objects, damage, weapons,
  │                                        │  surfaces, entry/exit points,
  │                                        │  anomalies → 2000 char report
  │                                        │         │
  └─────────────┬──────────────────────────┘         │
                │                                     │
                └──────────────┬──────────────────────┘
                               ▼
                         CHUNKING
               (size=500 chars, overlap=50)
                               │
                               ▼
                         EMBEDDING
               all-MiniLM-L6-v2 → 384-dim vector
               [0.12, -0.33, 0.78, ...]
                               │
                               ▼
                      ChromaDB COLLECTION
               Vectors stored for similarity search
               (isolated per case)
```

### Step 2: Why Groq Vision Instead of BLIP?

Previous versions used a local BLIP model which produced only generic captions like "a broken window" for every image.

We replaced it with **Groq Vision API** (`meta-llama/llama-4-scout-17b-16e-instruct`):

```
BLIP (old):
  "a broken window"  ← same 4 words for any crime scene image

Groq Vision (current):
  SCENE OVERVIEW: The image depicts a residential living room that has
  experienced significant disturbance. The space appears to have been
  ransacked, with furniture overturned...

  WEAPONS/INSTRUMENTS: A large kitchen knife (approximately 20-25cm blade)
  is visible on the floor near the couch, showing dark reddish-brown
  discoloration consistent with dried blood...

  EVIDENCE: Glass shards scattered near the window suggesting forced entry.
  Partial footprint visible in dust near the main entrance...
```

The richer description enables much more precise RAG retrieval.

### Step 3: Chunking (Why It Matters)

**Problem:** A 50-page PDF can't fit in an LLM's context window (8192 tokens ≈ 6000 words).

**Solution:** Split into small, overlapping chunks.

```
CHUNKING (size=500, overlap=50)

Chunk 1 (chars 0-499):
  "Officers arrived at the scene at 10:45 PM. The front door was forced open.
  Glass fragments were scattered across the entryway. A kitchen knife with
  blood traces was found on the living room floor..."

Chunk 2 (chars 450-949):    ← 50-char overlap keeps context continuous
  "...living room floor. The victim reported hearing a loud crash at
  approximately 10:30 PM. Neighbors confirmed unusual activity..."
```

### Step 4: Embeddings

An **embedding** converts text into a vector where similar meanings = similar vectors.

```
"knife"           → [0.82, -0.15, 0.43, 0.67, ...]  ─┐
"sharp blade"     → [0.80, -0.12, 0.45, 0.65, ...]   │  CLOSE (similar meaning)
"cutting weapon"  → [0.79, -0.18, 0.41, 0.69, ...]  ─┘

"chocolate cake"  → [-0.23, 0.89, -0.56, 0.12, ...]     FAR (different meaning)
```

**We use: `all-MiniLM-L6-v2`**
- Output: 384-dimensional vector
- Size: ~80MB (lightweight, cached after first download)
- Speed: ~14,000 sentences/second

### Step 5: ChromaDB Vector Search (Case-Isolated)

```
User question: "What weapon was found?"
                    │
                    ▼
Case resolution: session → active_case = "CASE-20260224-a1b2c3"
                    │
                    ▼
Question embedding: [0.78, -0.22, 0.55, 0.61, ...]
                    │
                    ▼
       ChromaDB SEARCH (Top-5, case-only)
  
  Chunk: "Officers arrived at 10:45 PM..."
  Distance: 15.7  ← far, NOT relevant

  Chunk: "A kitchen knife with blood traces..."
  Distance: 0.3   ← very close, HIGHLY relevant ✅

  Chunk: "Neighbors confirmed unusual activity..."
  Distance: 8.2   ← far, NOT relevant
                    │
                    ▼
         Return Top-5 closest chunks
         (ONLY from active case's collection)
```

**Distance Metric:**
```
L2 (Euclidean) Distance:  d = √(Σ(aᵢ - bᵢ)²)
Lower = more similar. We use ChromaDB with L2 space (same as old FAISS IndexFlatL2).
```

### Step 6: Context Injection & Generation

```
SYSTEM PROMPT (forensic analyst persona):
  "You are EvidenceAI. You think like a senior forensic investigator.
   Structure answers with: Summary, Evidence Details, Confidence Level..."

USER PROMPT:
  "CONTEXT:
   [Source 1 — PDF, Page 3] A kitchen knife with blood traces was found...
   [Source 2 — scene_image] ...knife visible on floor, dried blood
    discoloration consistent with type O blood, glass shards near window...
   [Source 3 — PDF, Page 1] Officers responded to burglary report...

   QUESTION: What weapon was found at the crime scene?"

LLM OUTPUT:
  "📋 SUMMARY: A kitchen knife is the primary weapon identified.

   🔍 EVIDENCE DETAILS:
   - Incident report (Page 3) confirms a kitchen knife with blood traces
     was recovered from the living room floor.
   - Scene image corroborates: knife approximately 20-25cm, blade shows
     dried reddish-brown discoloration.

   ⚠️ CONFIDENCE: High — consistent across both text report and visual evidence."
```

### Why RAG Prevents Hallucination

```
Without RAG:
  LLM uses training data → might fabricate "a gun was found" → HALLUCINATION

With RAG:
  LLM restricted to injected context → "The knife was found per report page 3"
  If context is silent → "Information not available in uploaded evidence"
```

### RAG vs Fine-Tuning

```
┌──────────────────┬─────────────────────────────────────────┐
│      RAG         │           Fine-Tuning                   │
├──────────────────┼─────────────────────────────────────────┤
│ Knowledge is     │ Knowledge baked into model weights      │
│ EXTERNAL (docs)  │                                         │
│ Update: add docs │ Update: retrain (expensive, slow)       │
│ Cost: ~$0        │ Cost: $100–$10,000 (GPU time)           │
│ No GPU needed    │ Needs powerful GPU                      │
│ Real-time data   │ Static (training cutoff date)           │
│ Best for: Q&A    │ Best for: teaching new skills/styles    │
│ on specific docs │                                         │
└──────────────────┴─────────────────────────────────────────┘
```

### Multimodal = Multiple Data Types

```
UNIMODAL (text only):
  PDF text → Chunk → Embed → ChromaDB → LLM → Answer

MULTIMODAL (text + images):
  PDF text ──────────→ Chunk → Embed ─┐
                                       ├──→ ChromaDB → LLM → Answer
  Image → Groq Vision → Text → Embed ─┘

The "bridge" between modalities:
  Image → Groq Vision API → Rich text description → Same embedding space as PDF
  This allows a text question to find relevant image information!
```

## �️ Advanced RAG Features

### 1. Case Supervisor (Auto-Linking & Isolation)

Prevents **cross-incident contamination** by automatically organizing uploads into isolated cases.

```
PROBLEM: Without case isolation
  Upload Case A (burglary) → All chunks in shared index
  Upload Case B (arson)    → All chunks in shared index
  Query about Case B → retrieves Case A evidence! ❌

SOLUTION: Case Supervisor + ChromaDB collections
  Upload Case A → Case Supervisor auto-creates CASE-001 → ChromaDB collection "CASE-001"
  Upload Case B → Case Supervisor auto-creates CASE-002 → ChromaDB collection "CASE-002"
  Query about Case B → searches ONLY "CASE-002" collection ✅
```

**Auto-linking algorithm:**
```
1. Upload arrives → extract text content
2. Groq LLM generates 300-char content summary
3. Summary → embed (384D vector)
4. Compare against ALL existing case embeddings (cosine similarity)
5. If similarity > 80% → link to that case (same incident)
6. If similarity ≤ 80% → create NEW case
7. Store in case-specific ChromaDB collection
```

**Session management:** Each browser session tracks an "active case". Queries only search within the active case. Users can switch cases manually via `/api/cases/select`.

### 2. XML-Based Jailbreak Defense

Prevents prompt injection attacks using a 3-layer defense system:

```
LAYER 1: Input Classification (regex)
  detect_jailbreak("ignore previous instructions") → BLOCKED ⚠️
  Cost: <1ms, blocks before any API call

LAYER 2: XML Prompt Wrapping
  <System> Highest authority — defines AI behavior </System>
  <RetrievedContext> Ground truth — forensic evidence </RetrievedContext>
  <UserQuery> Lowest priority — user's request only </UserQuery>

LAYER 3: Hardened System Prompt
  Explicit anti-injection rules embedded in system message
```

**Why XML tags work:** LLMs are trained on structured data (HTML/XML). They understand that content inside `<UserQuery>` tags is user text, not system instructions. Even if the user types "ignore previous instructions", the LLM recognizes it's inside `<UserQuery>` — not a system directive.

**Detected patterns:** instruction override, role manipulation, evidence tampering, system bypass, prompt leaking, encoding tricks, code injection.

### 3. HYDE (Hypothetical Document Embeddings)

Bridges the **query-document semantic gap** for 10-30% better recall.

```
WITHOUT HYDE:
  Question: "Was a weapon found?"          → embedding in QUESTION SPACE
  Document: "A kitchen knife was recovered" → embedding in DOCUMENT SPACE
  Distance: LARGE (different text styles)

WITH HYDE:
  Question: "Was a weapon found?"
  → LLM generates: "A sharp-edged weapon consistent with a kitchen
     knife was recovered from the crime scene..."
  → Embed THIS hypothetical answer
  → Distance to real document: SMALL (same text style!)
```

**Research basis:** "Precise Zero-Shot Dense Retrieval without Relevance Labels" (Gao et al., ACL 2023). HYDE outperforms standard dense retrieval by ~10-20% on multiple benchmarks.

### 4. Parent-Child Retrieval

Solves the **precision-context tradeoff** in chunking.

```
FLAT CHUNKING (old):
  Search precision: ★★★★☆ (500-char chunk matches well)
  Context richness:  ★★★☆☆ (LLM only sees 500 chars)

PARENT-CHILD (new):
  Search precision: ★★★★☆ (child 500-char embeddings for ChromaDB)
  Context richness:  ★★★★★ (parent full-page text sent to LLM)

  PDF Page → PARENT (full page, stored in metadata)
              ├── CHILD 1 (500 chars) → embedded in ChromaDB
              ├── CHILD 2 (500 chars) → embedded in ChromaDB
              └── CHILD 3 (500 chars) → embedded in ChromaDB

  Query → match CHILD 2 → return PARENT (full page) to LLM
```

### 5. RAG Evaluation Logger

Tracks quality metrics for every query and stores them in `evaluation_logs.json`.

| Metric | Description | Good Value |
|--------|-------------|------------|
| `retrieval_relevance` | Mean L2 distance of top-K chunks | < 0.5 |
| `grounding_score` | % of answer sentences grounded in context | > 70% |
| `answer_length` | Character count of generated answer | 500-3000 |
| `top_k_used` | Number of chunks actually retrieved | 5 |
| `hyde_used` | Whether HYDE was active for this query | true/false |
| `jailbreak_blocked` | Whether jailbreak guard blocked query | true/false |

View aggregate metrics via `GET /api/evaluation`.

### 6. Timeline Reconstruction

Extracts time-referenced events from case evidence and builds a **chronological timeline** using regex extraction + Groq LLM enrichment.

```
FLOW:
  All case chunks → Regex time extraction → Groq LLM enrichment → Sorted timeline

EXAMPLE OUTPUT:
  10:30 PM — Loud crash heard by neighbor at 221B West Ridge Apartments [PDF Page 1]
  11:15 PM — Complainant returned home, discovered break-in [PDF Page 2]
  11:25 PM — Complainant called police [PDF Page 2]
  11:45 PM — Officers arrived at the scene [PDF Page 3]

  GAPS: No events recorded between 10:30 PM and 11:15 PM (45-minute gap)
  CONTRADICTIONS: None detected
```

**Regex patterns detect:** 12-hour times (`10:30 PM`), 24-hour times (`22:30`), military format (`0830 hours`), written times (`midnight`, `noon`), approximate times (`around 11 PM`), time ranges (`between 10 and 11 PM`).

**LLM enrichment adds:** event descriptions, gap identification, contradiction detection, and confidence ratings (`high`/`medium`/`low`).

Access via `GET /api/timeline?session_id=default`

### 7. AI Hypothesis Generator (ACH)

Uses **Analysis of Competing Hypotheses** methodology to generate 3-5 investigative theories and rate each by evidence strength.

```
EXAMPLE OUTPUT:
  [STRONG]     Forced Entry Burglary (confidence: 0.85)
    ✓ Supporting: Broken window latch, footprints in mud, items stolen
    ✗ Contradicting: Glass found inside despite "outside" break-in
    ? Missing: Fingerprint analysis, CCTV footage review

  [MODERATE]   Staged Crime Scene / Insurance Fraud (confidence: 0.45)
    ✓ Supporting: Glass inside suggests break from inside, high-value items targeted
    ✗ Contradicting: Mud footprints suggest actual external entry
    ? Missing: Financial records, insurance claim history

  [WEAK]       Opportunistic Theft by Known Person (confidence: 0.25)
    ✓ Supporting: Targeted valuables suggest familiarity with layout
    ✗ Contradicting: No sign of key entry, forced entry evidence present
    ? Missing: Suspect interviews, neighbor witness statements
```

**Evidence strength ratings:** `strong` (0.70-1.00), `moderate` (0.40-0.69), `weak` (0.10-0.39), `speculative` (0.00-0.09).

Access via `POST /api/hypothesis` with body `{"session_id": "default"}`

### 8. Memory Across Sessions (Cross-Case Comparison)

Stores **detailed case summaries** persistently and enables **AI-powered cross-case comparison**.

```
STORE FLOW:
  Active case chunks → Groq LLM → Structured summary → case_memory.json
  Summary includes: type, location, time, method, evidence, findings, suspects, losses

COMPARE FLOW:
  Case A summary + Case B summary → Groq LLM → Comparative analysis
  Output: similarities, differences, patterns, possible connections, recommendations
```

**Use case:** Investigator uploads evidence for Case A (residential burglary, Feb 24), analyzes it, stores the summary. Weeks later, Case B arrives (another residential burglary, Mar 10). The investigator can ask: *"Compare this case with the previous one"* — and the AI identifies shared patterns (same time of night, same neighborhood, same method of entry) that might indicate a serial offender.

**Endpoints:**
- `GET /api/memory` — List all stored case summaries
- `POST /api/memory/store` — Generate & store summary for the active case
- `POST /api/memory/compare` — Compare two cases: `{"case_id_a": "...", "case_id_b": "..."}`

## 🎓 Concepts for Viva

**Q: What is RAG?**
> RAG (Retrieval-Augmented Generation) enhances LLM answers by first retrieving relevant documents from a knowledge base, then injecting them into the prompt. This grounds answers in real data and prevents hallucination.

**Q: Why not fine-tune the model?**
> Fine-tuning modifies model weights permanently and requires expensive GPU training. RAG is cheaper, updatable in real-time, and requires no retraining when data changes.

**Q: What makes this multimodal?**
> It processes multiple data types — text (PDF) and images. Images are analyzed by Groq Vision API (Llama 4 Scout) which generates rich forensic descriptions (~2000 characters), then these are embedded and stored alongside PDF text chunks in the same ChromaDB case collection.

**Q: Why did you switch from BLIP to Groq Vision?**
> The local BLIP model only generated generic 4-word captions like "a broken window" for every image regardless of content. Groq Vision API provides a 2000+ character detailed forensic analysis covering 7 categories (scene overview, objects, damage, weapons, surfaces, entry/exit points, anomalies) — far more useful for RAG retrieval.

**Q: How does ChromaDB work?**
> ChromaDB is a persistent vector database that stores embeddings in isolated collections — one per forensic case. Each collection independently stores 384-dimensional vectors, documents, and metadata. When a user queries, ChromaDB searches ONLY within the active case's collection using L2 distance (same math as FAISS IndexFlatL2). Unlike FAISS, ChromaDB natively supports persistence (survives restarts), metadata filtering, and document storage alongside vectors.

**Q: Why did you switch from FAISS to ChromaDB?**
> FAISS stores all vectors in a single flat index with no concept of case isolation. In a multi-case forensic system, evidence from a burglary case could leak into an arson case query (cross-incident contamination). ChromaDB provides native collection-level isolation — each case gets its own collection, and queries never cross collection boundaries. Additionally, ChromaDB handles persistence automatically while FAISS requires manual serialization.

**Q: Why chunking with overlap?**
> LLMs have limited context windows. We chunk large documents into 500-character pieces so: (1) each chunk creates a meaningful embedding, (2) we only retrieve what's relevant, (3) we stay within token limits. The 50-character overlap prevents critical sentences at chunk boundaries from being split.

**Q: What is temperature in LLMs?**
> Temperature controls output randomness. We use 0.3 — low enough for consistent, factual forensic answers while allowing natural language variation.

**Q: How does the theme toggle work?**
> A `data-theme` attribute is set on the `<html>` element. CSS variables defined under `[data-theme="light"]` and `[data-theme="dark"]` selectors override all colors globally. The preference is persisted in `localStorage` so it survives page refreshes.

**Q: Why does the intro video use `objectFit: contain`?**
> `contain` ensures the entire video is visible without cropping. `cover` would fill the screen but crop the top/bottom of the video, cutting off content. For an intro animation, showing the full frame is more important than filling every pixel.

**Q: What is HYDE and why does it improve retrieval?**
> HYDE (Hypothetical Document Embeddings) asks the LLM to generate a short hypothetical answer *before* searching. This hypothetical document is embedded instead of the raw query. Since the hypothetical doc uses the same language patterns as stored chunks (forensic terminology, sentence structure), the cosine/L2 similarity is significantly higher — improving recall by 15-30% in domain-specific RAG systems.

**Q: How does XML-based jailbreak protection work?**
> It's a 3-layer defense: Layer 1 uses 25+ regex patterns across 7 categories (role manipulation, prompt leak, encoding tricks, etc.) to detect injection attempts. Layer 2 wraps the prompt in XML tags (`<System>`, `<RetrievedContext>`, `<UserQuery>`) that enforce strict instruction hierarchy — the LLM treats `<System>` as highest priority. Layer 3 embeds explicit refusal instructions in the system prompt itself.

**Q: What is Parent-Child Retrieval?**
> Standard chunking creates small, fixed-size chunks (500 chars) for precise embedding matching, but the LLM may lack surrounding context. Parent-Child Retrieval searches on small child chunks for precision, then returns the full parent text (entire page) to the LLM. This eliminates context fragmentation — the LLM sees complete paragraphs while the search engine still benefits from fine-grained matching.

**Q: Why log RAG evaluation metrics?**
> Without evaluation, you can't tell if RAG is actually helping or hallucinating. Our evaluation module computes: (1) **Context Overlap** — what % of the answer's content comes from retrieved chunks (faithfulness proxy), (2) **Retrieval Relevance** — how close the retrieved chunks are to the query in vector space. These metrics are logged per query to `evaluation_logs.json` and aggregated via `/api/evaluation` for monitoring quality over time.

**Q: Can the advanced features be disabled?**
> Yes — each feature has an independent boolean flag in `config.py`: `USE_HYDE`, `ENABLE_JAILBREAK_GUARD`, `ENABLE_PARENT_CHILD`, `ENABLE_EVALUATION`, `ENABLE_AUTO_CASE_LINKING`, `ENABLE_TIMELINE`, `ENABLE_HYPOTHESIS`, `ENABLE_CASE_MEMORY`. Setting any to `False` reverts to the baseline behavior for that component. This makes A/B testing and debugging straightforward. All 8 features are fully modular.

**Q: What is Case Supervisor and how does auto-linking work?**
> The Case Supervisor module automatically organizes uploaded evidence into isolated cases. When a file is uploaded, the module: (1) Generates a content summary using Groq LLM, (2) Embeds the summary into a 384D vector, (3) Compares against all existing case embeddings using cosine similarity. If similarity > 80% (configurable), the upload is linked to the matching case. Otherwise, a new case is created. Each case maps to a separate ChromaDB collection, ensuring zero cross-contamination.

**Q: What happens if Case Supervisor is disabled?**
> When `ENABLE_AUTO_CASE_LINKING = False`, all uploads go to a single "default" collection (similar to how FAISS worked before the upgrade). Session/case management is bypassed, and the system behaves as a single-case RAG pipeline.

**Q: How does Timeline Reconstruction work?**
> It's a hybrid regex + LLM approach. First, 10+ regex patterns scan all case chunks for time mentions (12-hour, 24-hour, military format, written times like "midnight", approximate times like "around 11 PM"). Each match captures ±100 characters of surrounding context. Then all evidence is sent to Groq LLM with a structured timeline prompt. The LLM produces chronological events with descriptions, identifies time gaps (periods with no recorded activity), and flags contradictions between sources. This gives investigators a quick temporal overview of the incident.

**Q: What is Analysis of Competing Hypotheses (ACH)?**
> ACH is a structured analytical methodology developed by Richards Heuer for the CIA. Instead of picking the most "obvious" hypothesis and seeking confirming evidence (confirmation bias), ACH generates ALL plausible hypotheses and evaluates each against ALL evidence. The "winning" hypothesis isn't the one with the most supporting evidence — it's the one with the LEAST contradicting evidence. Our AI Hypothesis Generator approximates this by explicitly asking the LLM to identify both supporting AND contradicting evidence for each theory.

**Q: How does Memory Across Sessions enable cross-case comparison?**
> Each case gets a detailed summary stored in `case_memory.json` (separate from `case_registry.json` which is for auto-linking). Summaries capture: incident type, location, time, method of operation, key evidence, findings, suspects, and losses. When comparing, both summaries are sent to Groq LLM with a comparative analysis prompt. The LLM identifies: similarities (shared patterns), differences (distinguishing factors), behavioral/temporal patterns (same MO, same time of day), possible connections (serial offender indicators), and actionable recommendations (cross-reference fingerprints, interview witnesses from both cases).

**Q: Can the new features be disabled independently?**
> Yes — each has its own flag: `ENABLE_TIMELINE`, `ENABLE_HYPOTHESIS`, `ENABLE_CASE_MEMORY`. Setting any to `False` returns a disabled message from the endpoint without affecting other features. All 8 advanced features (HYDE, Jailbreak Guard, Parent-Child, Evaluation, Case Supervisor, Timeline, Hypothesis, Memory) are fully modular and independently toggleable.
